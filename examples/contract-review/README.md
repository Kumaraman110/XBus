# §6 — Public-beta dogfood scenario

A realistic, reproducible XBus collaboration between two Claude-Code-style
sessions — **not** a nonce echo. The task is a genuine engineering review: does
the proposed Orders API **v2.0.0** introduce breaking changes against the live
**v1.4.0** contract?

## Cast

- **architect** — owns the v2 proposal; wants a breaking-change review before shipping.
- **reviewer** — receives the contract over XBus, runs a real breaking-change
  analysis, and replies with a structured findings report.

## What happens (all over the secure XBUS-STP transport)

1. `architect` sends the full v2 OpenAPI contract (~2 KB) to `reviewer` as a
   `requires_ack` + `requires_reply` request, referencing the v1 baseline by repo path.
2. `reviewer` pulls its inbox — the request body is presented **exactly once** (§1).
3. `reviewer` runs `contract-diff.ts` (a real OpenAPI breaking-change analyzer)
   over v1 vs the received v2, then **acks acceptance**.
4. `reviewer` **replies** with a structured JSON report (verdict + per-finding table).
5. `architect` receives the **correlated** reply (matching correlationId / causationId).

## Files

| File | Role |
|------|------|
| `fixtures/orders-api-v1.json` | The live v1.4.0 contract (baseline). |
| `fixtures/orders-api-v2.json` | The proposed v2.0.0 contract under review. |
| `contract-diff.ts` | The real breaking-change analyzer (pure, deterministic). |
| `transcript.md` | The generated transcript of one run (regenerated by the test). |
| `../../tests/e2e/dogfood-contract-review.test.ts` | The harness that drives it. |

## Result

The analyzer finds **10 breaking changes** and returns verdict **BLOCK**, e.g.:

- `Order.status` → `state` and `Order.total_cents` → `Order.total` (object) —
  removed/renamed required fields (consumers reading them break).
- `OrderCreate` adds required `total` and `idempotency_key` — existing callers break.
- `OrderList` → `OrderPage` — the response wrapper schema was renamed.
- `listOrders ?limit` went from optional to **required**.

The recommendation in the reply: ship v2 as a **major** version with a migration
guide, not the proposed minor bump.

## Why this is a credible dogfood

- Real multi-KB payload delivered intact and shown once (§1 body-once invariant).
- Real domain work performed on the received content (not an echo).
- Full request → ack → correlated reply lifecycle over the encrypted transport.
- Reproducible: `npx vitest run tests/e2e/dogfood-contract-review.test.ts`
  regenerates `transcript.md`.
